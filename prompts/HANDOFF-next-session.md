## Інженерні уроки попередньої сесії (ВРАХУЙ у дизайні воркфлоу)

1. **Windows `nul`:** будь-який редирект у `/dev/null` створює биті файли `nul`, що ламають `git add`. Заборони агентам писати у `nul`/`/dev/null`; для типчека — `bunx tsc --noEmit` (без `--outfile`).
2. **Не паралель `bun install`:** дві паралельні інсталяції б'ються за `bun.lock`/`node_modules`. Якщо хвиля додає залежності — інсталюй раз, послідовно; решта агентів лише пишуть код.
3. **Не довіряй заявам агентів «tsc clean»:** вони запускають перевірку в різні моменти на рухомому дереві. ПІСЛЯ кожної хвилі САМ (як менеджер — це дозволена оркестрація) проганяй повний `bunx tsc --noEmit -p backend/tsconfig.json` і `bun run build` як справжній гейт.
4. **Реальний smoke ловить те, що tsc не бачить:** баг подвійного serve пройшов tsc, але валив рантайм. Тримай smoke (Docker Postgres + curl ендпоінтів) як фінальний гейт хвилі.
5. **Дисципліна спільних файлів:** агенти в одній паралельній хвилі НЕ повинні редагувати ті самі файли. Спільні точки: `backend/src/routes/api.ts` (маунти), `backend/src/index.ts` (boot), `backend/src/db/schema.ts`, `frontend/src/locales/*.json`, `frontend/src/App.vue`. Редактори цих файлів — послідовно або один власник.
6. **worktree-ізоляція** (`isolation:'worktree'`) — лише якщо агенти справді мутують одне дерево конфліктно; для залежних правок не підходить (наступний агент не побачить файлів попереднього).

# Statok — Функціональні вимоги (Фаза 1)

> Чернетка функціональної частини ТЗ. Цільова аудиторія — агент, що **автономно** імплементує систему.
> Джерела істини: `specs/requirements.md` (рішення власника) — НЕ відхилятися; `specs/_draft-data-arch.md` (модель даних, API, сервіси) — УЗГОДЖУВАТИСЯ в назвах сутностей/полів/ендпоінтів.
> Схема БД, ендпоінти, деплой, CI/CD — НЕ дублюються тут. Посилання на архітектурний документ: `arch §N`.
> Функціональна вимога = `FR-NN` + acceptance criteria (тестовні пункти `- [ ]`). Усі суми у форматі `*Minor` (integer minor units) + код валюти; кількості/ціни/курси — `numeric` (string у JS); базова валюта — `BASE_CURRENCY` (USD).

## Глосарій і наскрізні конвенції

| Термін | Значення |
|---|---|
| Позиція (holding) | Похідна величина: кількість активу на (рахунок, актив), згорнута з транзакцій. Окремої таблиці нема (`arch §0`, `§3.1`). |
| Кеш-залишок | Похідна величина: сальдо по (рахунок, валюта), згорнуте з грошових транзакцій. |
| Cash-актив | Технічний актив `type='cash'`, `symbol = currency` (напр. USD). Створюється автоматично `ensureCashAsset` (`arch §1.4`). Ціна тотожно 1. |
| Собівартість (cost basis) | Середньозважена (average cost) сукупна вартість придбання позиції у валюті активу, включно з комісіями (`arch §3.1`). |
| Базова валюта | `BASE_CURRENCY` (USD). Усі звіти/тотали конвертуються в неї; позиції живуть у власних валютах. |
| Бізнес-дата | `date`, обчислена у TZ `Europe/Kyiv` (`arch §0`). |
| EOD-пайплайн | Денна джоба 23:30 Kyiv: ціни → курси → погашення облігацій → снапшот (`arch §4`). |
| minor units | Найменша одиниця валюти (копійка/цент). Формат сум у БД та API. |

**Наскрізні правила (стосуються всіх FR, далі не повторюються):**

- **CRR-1 (auth-gate):** усі ендпоінти, крім `POST /auth/login` і `GET /health`, доступні лише з валідним Bearer JWT; без токена → 401 `UNAUTHORIZED`.
- **CRR-2 (формат помилок):** будь-яка помилка → `{ "error": "MACHINE_CODE", "message": "..." }` з кодами статусів за `arch §2`.
- **CRR-3 (гроші):** жодних обчислень через float; лише decimal-хелпери `packages/shared` (`arch §0`). Округлення half-up до minor unit на межі numeric→minor.
- **CRR-4 (валюти):** код валюти — ISO-4217, 3 літери, верхній регістр. Невідомий/невалідний код → 400 `VALIDATION_ERROR`.
- **CRR-5 (single-user):** усі доменні записи належать сід-користувачу (`user_id`); крос-юзерної видимості нема.
- **CRR-6 (i18n):** усі тексти UI — через vue-i18n (`uk` дефолт, `en`), без хардкоду рядків у компонентах (`arch §7.3`).
- **CRR-7 (мобільний адаптив):** кожен екран коректно відображається на ширині від 360px (одноколонкова розкладка, без горизонтального скролу контенту).

## 1. Автентифікація

Логін/пароль із сід-адміна (env), JWT (`jose`, HS256, TTL 7 діб), rate-limit логіну. Деталі реалізації — `arch §1.2`, `§2 auth`, `§9`.

### FR-01 — Сід адміністратора з env
Єдиний користувач створюється автоматично при старті бекенда з `ADMIN_USERNAME`/`ADMIN_PASSWORD`.

- [ ] При першому старті з порожньою таблицею `users` створюється користувач із `username = ADMIN_USERNAME`, `password_hash = bcrypt(ADMIN_PASSWORD, 10)`.
- [ ] При повторному старті, якщо користувач із таким `username` уже існує, сід НЕ перезаписує його (пароль не змінюється).
- [ ] Зміна `ADMIN_PASSWORD` в env + перезапуск НЕ змінює пароль наявного користувача (зміна пароля у v1 — лише вручну SQL/пересідом, документується в README).
- [ ] Відсутній `JWT_SECRET` або `DATABASE_URL` на старті → процес завершується з fatal-помилкою і зрозумілим повідомленням (`arch §8.3`).

### FR-02 — Логін і видача токена
`POST /auth/login {username, password}` → `{token, username}`.

- [ ] Валідні креденшали → 200 із JWT (claims `sub`, `username`, `exp`; TTL 7 діб).
- [ ] Невірний пароль або неіснуючий username → 401 `UNAUTHORIZED` з однаковим повідомленням (не розкривати, що саме невірне).
- [ ] У лог пишеться лише `username` + `ip` + причина; пароль/токен у логи НЕ потрапляють.
- [ ] Тіло без `username` або `password` → 400 `VALIDATION_ERROR`.

### FR-03 — Rate-limit логіну
In-memory лічильник невдалих спроб по IP (`x-forwarded-for` від Traefik).

- [ ] Після 5 невдалих спроб за вікно 15 хв з одного IP наступна спроба → 429 `RATE_LIMITED` із заголовком `Retry-After`.
- [ ] Успішний логін скидає лічильник для цього IP.
- [ ] Вікно ковзне/скидається після 15 хв без перевищення; легітимний користувач після паузи може зайти.
- [ ] Rate-limit НЕ застосовується до інших ендпоінтів (лише `/auth/login`).

### FR-04 — Сесія, refresh, вихід
Sliding re-issue; ревокації на сервері нема (`arch §9`).

- [ ] `POST /auth/refresh` зі ще валідним токеном → новий токен зі свіжим TTL 7 діб.
- [ ] Фронтенд викликає `refresh` при старті застосунку (якщо токен у localStorage є й не протух).
- [ ] `GET /auth/me` → `{userId, username}` для валідного токена.
- [ ] Протермінований/невалідний токен на будь-якому `/api/*` → 401; фронтенд при 401 чистить токен і редіректить на `/login`.
- [ ] `POST /auth/logout` → `{ok:true}` (лог-запис); інвалідація фактично клієнтська (видалення токена з localStorage `statok_token`).

## 2. Рахунки

CRUD рахунків; типи `broker/exchange/bank/wallet/other`; рахунок **мультивалютний** — кеш-залишки по валютах виводяться з транзакцій (`arch §1.3`, `§2 accounts`).

> Канон типів (`account_kind`): `broker`, `exchange`, `bank`, `wallet`, `other`. Власникові «cash/готівка» моделюється як рахунок `kind='other'` (або `bank`) із cash-залишками; окремого `kind='cash'` нема — готівка/депозит представлені cash-**активом** відповідної валюти на рахунку.

### FR-05 — Створення рахунку
`POST /api/accounts {name, kind, note?}`.

- [ ] Створюється рахунок із `name`, `kind ∈ {broker,exchange,bank,wallet,other}`, опційним `note`; дефолт `kind='broker'`.
- [ ] `name` унікальний у межах користувача; дубль → 409 (імʼя зайняте).
- [ ] Порожній `name` або `kind` поза переліком → 400 `VALIDATION_ERROR`.
- [ ] Новий рахунок одразу має порожній набір балансів (жодних транзакцій).

### FR-06 — Перегляд рахунків із балансами
`GET /api/accounts?withBalances=true&includeArchived=`.

- [ ] Без параметрів повертає список активних (не архівних) рахунків.
- [ ] `withBalances=true` додає для кожного рахунку `balances:[{currency, cashMinor}]` (по кожній валюті з ненульовим або наявним кеш-рухом) і `valueBaseMinor` (повна вартість рахунку = кеш + позиції, конвертована в базову).
- [ ] `includeArchived=true` додає архівні рахунки; без нього — приховані.
- [ ] Мультивалютний рахунок показує кілька рядків балансу (напр. UAH, USD, EUR) — по одному на валюту.
- [ ] Сортування — за `sortOrder` (зрост.), далі за `name`.

### FR-07 — Редагування рахунку
`PUT /api/accounts/:id {name?, kind?, note?, sortOrder?, archived?}`.

- [ ] Часткове оновлення будь-якого з полів `name/kind/note/sortOrder`.
- [ ] Перейменування на зайняте імʼя → 409.
- [ ] `archived=true` архівує рахунок (ставить `archivedAt`); `archived=false` розархівовує.
- [ ] Архівний рахунок не зникає з історії транзакцій і зі звітів; лише ховається зі стандартних списків вводу.
- [ ] Неіснуючий `:id` → 404.

### FR-08 — Видалення/архівація рахунку
`DELETE /api/accounts/:id`.

- [ ] Рахунок без жодної транзакції видаляється (hard delete) → 204.
- [ ] Рахунок із транзакціями → 409 `ACCOUNT_HAS_TRANSACTIONS`; UI пропонує архівацію замість видалення.
- [ ] Після відмови у видаленні рахунок лишається без змін.

### FR-09 — Мультивалютні кеш-залишки рахунку
Похідні від грошових транзакцій (deposit/withdraw/transfer/dividend/coupon/interest/buy/sell/opening_balance).

- [ ] Для рахунку власника з рухами в UAH/USD/EUR кеш показується трьома окремими залишками.
- [ ] Кеш-залишок у валюті = сума всіх грошових дельт цієї валюти на рахунку (`arch §3.1`); знак коректний (поповнення +, списання −).
- [ ] Негативний кеш-залишок ДОЗВОЛЕНИЙ (показується з warning у UI, не блокується) — наприклад покупка «в борг» до поповнення.
- [ ] Залишок з нульовим підсумком, але наявними рухами, може показуватись як 0.00 (не приховується помилково).

## 3. Активи

Каталог активів; типи `stock/etf/crypto/bond/cash`; для `bond` — `bond_details` (`arch §1.4`, `§1.5`, `§2 assets`). `cash`-активи створюються автоматично, вручну не заводяться.

### FR-10 — Створення активу
`POST /api/assets {type, symbol, name?, currency, priceSource?, bond?}`.

- [ ] Створюється актив типу `stock/etf/crypto/bond` із `symbol`, `currency`, опційними `name`, `priceSource`.
- [ ] `symbol` унікальний у межах (користувач, type); дубль → 409.
- [ ] Семантика `symbol` за типом: stock/etf — Yahoo-тікер (`AAPL`, `VWRA.L`); crypto — Yahoo-пара (`BTC-USD`); bond — ISIN (`UA4000227696`).
- [ ] `priceSource` дефолт `yahoo` для stock/etf/crypto; для `bond` дефолт `manual`.
- [ ] Спроба створити `type='cash'` вручну через цей ендпоінт → 400 (cash-активи лише автоматичні).
- [ ] Невалідна валюта (не ISO-4217) → 400.

### FR-11 — Bond-деталі при створенні/редагуванні
Для `type='bond'` блок `bond` обовʼязковий; зберігається атомарно з активом (`arch §1.5`).

- [ ] `type='bond'` без блоку `bond` → 400; `bond` для не-bond типу → 400.
- [ ] `bond` містить `faceValueMinor` (>0), `couponRatePercent`, `couponFrequency ∈ {0,1,2,4,12}`, `maturityDate`; опційно `issueDate`, `isin`.
- [ ] `couponFrequency=0` (zero-coupon) допускається лише з `couponRatePercent=0`, і навпаки (узгоджений CHECK `arch §1.5`); порушення → 400.
- [ ] Asset + bond_details створюються в одній DB-транзакції; помилка валідації відкочує обидва.
- [ ] `GET /api/assets/:id` для облігації повертає вкладений блок `bond` з усіма полями.

### FR-12 — Перегляд і редагування каталогу активів
`GET /api/assets?type=&includeArchived=`; `PUT /api/assets/:id`.

- [ ] Список фільтрується за `type`; `includeArchived=true` показує архівні.
- [ ] `PUT` оновлює `name/currency/priceSource/archived/bond`; `symbol` через PUT НЕ змінюється (лише через `ticker_change`, `arch §1.4`).
- [ ] Спроба змінити `symbol` через PUT → ігнорується або 400 (поле не приймається).
- [ ] Архівація активу не видаляє його транзакції та історію цін.

### FR-13 — Видалення активу
`DELETE /api/assets/:id`.

- [ ] Актив без транзакцій видаляється → 204 (каскадно зносяться його `price_quotes` і `bond_details`).
- [ ] Актив із транзакціями → 409 `ASSET_HAS_TRANSACTIONS`; пропонується архівація.
- [ ] Cash-актив, на який посилаються грошові транзакції, видалити не можна (409).

## 4. Транзакції

CRUD усіх типів; повна матриця полів і CHECK-и — `arch §1.6`; ендпоінти — `arch §2 transactions`. Тип транзакції НЕ змінюється після створення.

> Канонічні типи (`transaction_type`): `buy`, `sell`, `deposit`, `withdraw`, `transfer_out`, `transfer_in`, `dividend`, `coupon`, `interest`, `split`, `ticker_change`, `opening_balance`. Комісія — поле `feeMinor` на `buy`/`sell`, НЕ окремий тип.

### FR-14 — Купівля (buy)
`buy`: списує кеш, нарощує позицію; комісія включається в собівартість.

- [ ] `buy {accountId, assetId, quantity, price, amountMinor, currency, feeMinor?}`: `amountMinor` має дорівнювати `quantity × price` (без комісії), інакше 400.
- [ ] `currency` транзакції має дорівнювати `assets.currency`; розбіжність → 400 `CURRENCY_MISMATCH`.
- [ ] Собівартість позиції зростає на `amountMinor + feeMinor`; кеш валюти зменшується на `amountMinor + feeMinor` (`arch §3.1`).
- [ ] `quantity > 0` і `price` задані; відсутність — 400.
- [ ] Дробова кількість підтримується (дробові акції/крипта; до 18 знаків).

### FR-15 — Продаж (sell)
`sell`: зменшує позицію за середньою собівартістю, фіксує realized P&L; комісія зменшує виручку.

- [ ] `sell` із `quantity`, `price`, `amountMinor`; кеш зростає на `amountMinor − feeMinor`.
- [ ] Реалізований результат = `(amountMinor − feeMinor) − costPart`, де `costPart = roundHalfUp(costBasis × quantity / qtyHeld)` (`arch §3.1`).
- [ ] Спроба продати більше, ніж є на (рахунок, актив) у будь-якій точці таймлайну → 409 `INSUFFICIENT_QUANTITY` (реплей таймлайну, `arch §1.6`).
- [ ] Собівартість позиції зменшується на `costPart`; кількість — на `quantity`.
- [ ] Продаж усієї позиції доводить `qty` до 0; позиція зникає зі списку, але її realized лишається у звіті P&L.

### FR-15a — Перевірка достатності кількості (наскрізна)
Реплей хронології `(account_id, asset_id)` при кожному записі/редагуванні/видаленні.

- [ ] Будь-яка операція (`sell`, `split` з дробовим множником, видалення `buy`, редагування `quantity`), що робить qty < 0 у будь-якій історичній точці → 409 `INSUFFICIENT_QUANTITY`, зміна не застосовується.
- [ ] Негативний КЕШ при цьому НЕ блокується (лише негативна кількість активу).
- [ ] Повідомлення помилки вказує актив/рахунок/дату конфлікту (human-readable `message`).

### FR-16 — Грошові операції (deposit / withdraw)
Поповнення та виведення кешу рахунку; `assetId` опційний у запиті — сервер сам резолвить cash-актив валюти.

- [ ] `deposit {accountId, amountMinor, currency, executedAt}` (без `assetId`) → сервер викликає `ensureCashAsset(currency)`, кеш валюти зростає.
- [ ] `withdraw` — симетрично, кеш зменшується.
- [ ] Якщо cash-активу цієї валюти ще нема, він створюється автоматично (`symbol=currency`, `priceSource='manual'`).
- [ ] `amountMinor` від'ємний або нуль → 400 (сума операції — модуль, знак визначає тип).

### FR-17 — Перекази між рахунками (строго парою)
`POST /api/transactions/transfer` — атомарна пара `transfer_out`/`transfer_in` зі спільним `transferGroupId` (`arch §1.6`, `§2`).

- [ ] `{fromAccountId, toAccountId, executedAt, outAmountMinor, outCurrency, inAmountMinor, inCurrency, note?}` створює ДВА рядки в одній DB-транзакції зі спільним `transferGroupId`.
- [ ] Валюта і сума «ноги» виходу та «ноги» входу НЕЗАЛЕЖНІ — це покриває і простий переказ (однакові), і конвертацію валюти між рахунками (різні).
- [ ] Перекази НЕ впливають на грошові потоки/дохідність (внутрішнє переміщення; виключені з cashflow-звіту, `arch §2 dashboards`).
- [ ] Видалення будь-якої ноги видаляє всю пару (`arch §2`).
- [ ] Редагування ноги: `executedAt`/`note` синхронізуються на обидві; сума/валюта — пер-нога (`arch §1.6`).
- [ ] `fromAccountId == toAccountId` → 400.
- [ ] In-kind переказ активів (не кешу) — НЕ підтримується у v1 (лише грошові ноги).

### FR-18 — Дивіденди / купони / відсотки (брутто / податок / нетто)
`dividend` (stock/etf), `coupon` (bond), `interest` (cash) — поля `grossMinor`, `withholdingTaxMinor`, `netMinor` (`arch §1.6`).

- [ ] Запис вимагає `grossMinor` і `withholdingTaxMinor`; `netMinor` має дорівнювати `grossMinor − withholdingTaxMinor` (узгоджений CHECK); порушення → 400.
- [ ] `withholdingTaxMinor` дефолт 0, якщо не задано (для дивідендів).
- [ ] `dividend` лише для stock/etf, `coupon` лише для bond, `interest` лише для cash-активу; інакше 400.
- [ ] Виплата нараховує `netMinor` на кеш відповідної валюти; позицію/собівартість не змінює (`arch §3.1`).
- [ ] Виплати потрапляють у звіт грошових потоків і у блок income звіту P&L.

### FR-19 — Корпоративні дії: спліт і зміна тікера (низький пріоритет)
`split` (множник кількості) і `ticker_change` (зі збереженням історії) — `arch §1.6`, `§3.1`.

- [ ] `split {assetId, quantity (множник), executedAt}`: нова кількість = стара × `quantity`; собівартість незмінна (середня ціна масштабується неявно).
- [ ] Reverse split — дробовий множник (напр. 0.1); якщо це робить qty < 0 — неможливо (qty лишається додатною при множенні на >0), але дробові залишки допускаються.
- [ ] `ticker_change` через `POST /api/transactions/ticker-change {assetId, newSymbol, executedAt}`: атомарно інсертить транзакцію з `meta {fromSymbol,toSymbol}` і оновлює `assets.symbol`.
- [ ] Історія цін лишається на тому ж `asset_id` (континуїтет); математика позиції не зачіпається.
- [ ] `newSymbol` зайнятий іншим активом → 409.
- [ ] `split`/`ticker_change` для cash-активу → 400.
- [ ] Видалення `ticker_change` відкочує `symbol`, якщо це останній `ticker_change` активу.

### FR-20 — Opening balance: механізм «поточного стану»
Завести наявну позицію або кеш-залишок без повної історії угод (`arch §1.6` матриця, `§3.1`). Вирішення відкритого питання requirements.md №1 — нижче.

- [ ] **opening_balance (актив):** `{accountId, assetId, quantity, amountMinor?, price?}` — заводить позицію: `qty += quantity`, `costBasis += amountMinor` (сукупна собівартість).
- [ ] Якщо `amountMinor` не задано → собівартість = `roundHalfUp(quantity × останнє котирування ≤ дати)`; якщо котирування немає взагалі → собівартість 0 і позиція позначається `costBasisIncomplete: true` у DTO (UI показує warning).
- [ ] **opening_balance (кеш):** `{accountId, currency, amountMinor}` (assetId опційний → cash-актив валюти) — стартовий залишок: `cash[currency] += amountMinor`.
- [ ] `opening_balance` для cash-активу з полем `quantity` → трактується як кеш-варіант (quantity ігнорується/заборонено).
- [ ] Opening balance НЕ створює парних транзакцій і НЕ впливає на cashflow-звіт (це стартовий стан, не операція внеску).
- [ ] **UX-механіка (рішення):** реалізується як окремий тип транзакції `opening_balance` (не окремий майстер-стейт), доступний з форми додавання транзакції та як швидка дія «Завести наявну позицію/залишок» на екрані рахунку. Кілька opening_balance на один (рахунок, актив) сумуються (хоча типово один).

### FR-21 — Журнал транзакцій із фільтрами та пагінацією
`GET /api/transactions?accountId=&assetId=&type=&from=&to=&limit=&offset=` (`arch §2`).

- [ ] Список сортується `executedAt` спадно; відповідь `{items, total}`.
- [ ] Фільтри `accountId`, `assetId`, `type`, діапазон дат `from`/`to` комбінуються (AND).
- [ ] Пагінація: `limit` дефолт 50, макс 500; `offset`; `total` відображає повну кількість під фільтром.
- [ ] Невалідний `type`/дата → 400.
- [ ] Кожен елемент містить достатньо для відображення рядка журналу (тип, дата, рахунок, актив-символ, кількість/сума/валюта, комісія, нетто для виплат).

### FR-22 — Перерахунок похідних при редагуванні/видаленні
Редагування/видалення транзакції → похідні (позиції, кеш, P&L) перераховуються (fold на льоту, `arch §3.1`).

- [ ] `PUT /api/transactions/:id` оновлює поля того самого типу; зміна `type` заборонена (400).
- [ ] Після будь-якої зміни наступні запити positions/valuation/pnl віддають перераховані значення (кеш не зберігається — fold щоразу).
- [ ] Редагування/видалення, що порушує інваріант кількості, → 409 `INSUFFICIENT_QUANTITY`, зміна відкочується.
- [ ] Історичні `net_worth_snapshots` НЕ перебудовуються автоматично; для перерахунку — явні `POST /api/snapshots/run|rebuild` (`arch §1.9`, FR-44).
- [ ] Видалення → 204; неіснуючий id → 404.

## 5. Облігаційний модуль (НАЙВИЩИЙ ПРІОРИТЕТ)

Власник зараз тримає лише ОВДП. Купонний розклад, очікувані виплати, фіксація купона, автопогашення, YTM/поточна дохідність. Сервіс — `arch §3.3`; ендпоінти — `arch §2 assets` (`/bond/schedule`, `/bond/metrics`).

### FR-23 — Генерація купонного розкладу
`GET /api/assets/:id/bond/schedule` → купони + фінальний рядок погашення (`arch §3.3`).

- [ ] Розклад генерується від `maturityDate` назад кроком `12 / couponFrequency` місяців до `issueDate` (або до найранішої транзакції активу, якщо `issueDate` NULL; ліміт 50 років).
- [ ] Сума купона = `roundHalfUp(faceValueMinor × couponRatePercent / 100 / couponFrequency)`, у валюті активу.
- [ ] Кожен рядок має `date`, `amountMinor`, `isFuture` (дата > сьогодні Kyiv).
- [ ] Фінальний рядок — погашення номіналу на `maturityDate` (`faceValue`).
- [ ] Zero-coupon (`couponFrequency=0`) → купонів нема, лише рядок погашення.
- [ ] Розклад read-only (обчислюваний); запит на не-bond актив → 404.

### FR-24 — Очікувані майбутні виплати
Майбутні купони (і погашення) виділяються у розкладі для планування.

- [ ] Рядки з `isFuture=true` дають перелік очікуваних надходжень із датами і сумами.
- [ ] Сума майбутніх купонів за період обчислювана на фронтенді з розкладу (агрегат не вимагає окремого ендпоінта).
- [ ] Уже виплачені (минулі) купони відрізняються від майбутніх прапором `isFuture`.

### FR-25 — Фіксація купона
Фактичний купон вноситься користувачем як транзакція `coupon` із податком (`arch §3.3`, FR-18).

- [ ] Користувач може записати `coupon`-транзакцію (брутто/податок/нетто) — типово зі значень розкладу, але суми редаговані вручну (фактична виплата може відрізнятись).
- [ ] Записаний купон нараховує `netMinor` на кеш валюти облігації.
- [ ] Розклад лишається обчислюваним і НЕ перетворюється на транзакції автоматично (фіксація — ручна дія користувача).
- [ ] (UI-зручність) Кнопка «записати купон» із рядка розкладу префілить форму `coupon` — деталі UI поза цим FR, але дані беруться з FR-23.

### FR-26 — Автопогашення в кеш на дату maturity
`processMaturedBonds(today)` — авто-`sell` усього залишку за номіналом на `maturityDate` (`arch §3.3`).

- [ ] Для облігації з `maturityDate ≤ today` і залишком qty > 0 створюється `sell`: `quantity = весь залишок`, `price = faceValue`, `amountMinor = roundHalfUp(qty × faceValue)`, `fee 0`, `executedAt = maturityDate 12:00 Kyiv`, `meta {autoRedemption:true}`.
- [ ] Кеш валюти облігації зростає на суму погашення; позиція доводиться до 0.
- [ ] **Ідемпотентність:** якщо залишок уже 0 (погашення вже відбулось) — крок пропускається, дубль не створюється.
- [ ] Викликається в EOD-пайплайні та при ручному `POST /api/snapshots/run` (`arch §3.3`, `§4`).
- [ ] Автопогашення позначене в `meta` (відрізнити від ручного продажу).

### FR-27 — YTM і поточна дохідність
`GET /api/assets/:id/bond/metrics?price=&date=` → `{ytmPercent, currentYieldPercent, priceUsed, priceSource, asOf}` (`arch §3.3`).

- [ ] Поточна дохідність = `(faceValue × couponRatePercent%) / cleanPrice` (річний купон / поточна ціна).
- [ ] YTM — ітеративний Newton–Raphson по рівнянню ціни (fallback бісекція), ACT/365F від `settlementDate`; повертає річний відсоток (`arch §3.3`).
- [ ] `price` дефолт = останнє котирування активу; за відсутності котирувань fallback — номінал (`faceValue`).
- [ ] Відповідь вказує `priceUsed`, `priceSource` (yahoo/manual/face) і `asOf` (дата ціни).
- [ ] Для zero-coupon currentYield = 0; YTM рахується лише з дисконту до номіналу.
- [ ] Запит на не-bond актив → 404.

## 7. Курси валют

Денні Frankfurter/ECB + НБУ (UAH); історія в БД; конвертація всього в базову; fallback — останній наявний курс. Джоба — `arch §4.2`; сервіс — `arch §3.4`; таблиця — `arch §1.8`; ендпоінти — `arch §2 fx`.

### FR-32 — Денний автозбір курсів (Frankfurter + НБУ)
Дві незалежні гілки в EOD-пайплайні (`arch §4.2`).

- [ ] Frankfurter тягне крос-курси ECB із базою USD для потрібних валют (`distinct(assets.currency) ∪ {BASE_CURRENCY, UAH, USD, EUR}` мінус USD/UAH).
- [ ] НБУ тягне офіційний курс UAH (UAH за 1 одиницю валюти) для потрібних валют.
- [ ] Курси зберігаються нормалізовано: Frankfurter → `(USD, ccy, rate, 'frankfurter')`; НБУ → `(ccy, UAH, rate, 'nbu')` (`arch §1.8`).
- [ ] Фейл однієї гілки не блокує іншу (Frankfurter не має UAH — НБУ обовʼязковий).
- [ ] Один рядок на (дата, base, quote); upsert; стан → `app_settings['job.fx']`.
- [ ] У вихідні Frankfurter віддає дату останньої пʼятниці — upsert саме тієї дати (натуральний fallback).

### FR-33 — Конвертація в базову валюту з fallback
`convert(amountMinor, from, to, date)` (`arch §3.4`); `GET /api/fx/convert`.

- [ ] `from === to` → identity (без курсу).
- [ ] Резолвинг курсу: прямий рядок із максимальним `rate_date ≤ date` (fallback «останній попередній»); інакше зворотний (`1/rate`); інакше pivot через USD (один рівень).
- [ ] Жодного курсу для валюти → 404 `FX_RATE_NOT_FOUND` (в API) / фейл джоби з лог-записом (у снапшоті).
- [ ] Відповідь `convert` вказує `rateUsed` і `rateDate` (≤ запитаної дати — видно застосований fallback).
- [ ] Конвертація — через bigint fixed-point (rate scale 8), round half-up; без float.
- [ ] Усі тотали/звіти конвертуються в `BASE_CURRENCY` цим сервісом.

### FR-34 — Ручне редагування курсу та історія
`PUT /api/fx/:date/:base/:quote {rate}` (upsert manual); `GET /api/fx?base=&quote=&from=&to=` (`arch §2`).

- [ ] Ручний курс — upsert по (дата, пара) із `source='manual'`; автозбір не перетирає manual-рядки.
- [ ] Manual-курси беруть участь у резолвингу нарівні з авто.
- [ ] Історія курсів за діапазоном повертається з `source` кожного рядка.
- [ ] Денна історія НБУ зберігається в БД (двері для майбутнього річного звіту в UAH без міграції, `requirements.md`).
- [ ] `POST /api/fx/sync` — ручний тригер; відповідь `{frankfurter:{ok,ratesUpserted}, nbu:{ok,ratesUpserted}}`.

## 8. Оцінка портфеля і P&L

Середньозважена собівартість; realized/unrealized (абсолют і %); розрізи по рахунку/активу/портфелю; усе в базовій. Сервіси — `arch §3.1`, `§3.2`; ендпоінти — `arch §2 portfolio`.

### FR-35 — Позиції з поточною оцінкою
`GET /api/portfolio/positions?accountId=&date=` (`arch §2`, `§3.1`).

- [ ] Кожна позиція: `quantity`, `costBasisMinor`, `avgCostMinor` (= `roundHalfUp(costBasis/qty)`), `lastPrice`+`priceDate`, `valueMinor` (= `qty × lastPrice`), `valueBaseMinor`, `unrealizedMinor`, `unrealizedBaseMinor`, `unrealizedPct`.
- [ ] `lastPrice` — останнє котирування `quote_date ≤ date`; для bond без котирування fallback — номінал; для stock/etf/crypto без котирування — `valueMinor: null` (у тотали не входить, UI показує warning).
- [ ] Позиції з `qty=0` не показуються.
- [ ] Окремий блок `cash:[{accountId, currency, balanceMinor, balanceBaseMinor}]`.
- [ ] `accountId` фільтрує до одного рахунку; без нього — весь портфель.
- [ ] `unrealizedPct` = `unrealized / costBasis` (коректний для додатної собівартості; при costBasis 0 — null або 0).

### FR-36 — Оцінка портфеля (valuation) з розрізами
`GET /api/portfolio/valuation?date=` → `{totalBaseMinor, byClass, byAccount, byCurrency, ...}` (`arch §2`).

- [ ] Тотал портфеля в базовій валюті на дату = сума value всіх позицій + кеш, конвертовано.
- [ ] Розріз `byClass` (stock/etf/crypto/bond/cash), `byAccount`, `byCurrency` — усі значення в базовій валюті.
- [ ] Структура збігається з `breakdown` снапшота (`arch §1.9`) — один формат для дашборда і снапшотів.
- [ ] `date` дефолт — сьогодні (Kyiv); історична дата дає оцінку на той день за збереженими цінами/курсами.
- [ ] Немає жодного курсу для валюти позиції → 404 `FX_RATE_NOT_FOUND`.

### FR-37 — P&L (realized + unrealized + income + fees)
`GET /api/portfolio/pnl?accountId=&from=&to=` (`arch §2`, `§3.2`).

- [ ] **Realized trading** за `[from, to]`: fold по повній історії, але акумулюються лише sell-и з `executedAt ∈ [from,to]`; конвертація за курсом дати кожного sell.
- [ ] **Income**: сума `netMinor` по dividend/coupon/interest за період (окремі підсумки за типами), конвертація за курсом дати виплати.
- [ ] **Fees**: сума `feeMinor` за період (довідково; вже враховані в собівартості/виручці).
- [ ] **Unrealized**: з поточного стану за поточним курсом.
- [ ] `totalBaseMinor = realizedTrading + incomeTotal + unrealized`.
- [ ] Розріз `perAsset` із realized/income/unrealized по кожному активу (у валюті активу і в базовій).
- [ ] Абсолютні суми і відсотки коректні; розрізи по рахунку через `accountId`.

### FR-38 — P&L у відсотках
Відсоткові показники дохідності позицій/портфеля.

- [ ] Unrealized % позиції = `unrealizedMinor / costBasisMinor` (FR-35).
- [ ] Realized % (за бажанням на рівні активу) = `realized / costPart` — обчислюваний із даних perAsset.
- [ ] XIRR/TWR/бенчмарки — ПОЗА скоупом v1 (лише проста P&L, `requirements.md`).

## 9. Net worth снапшоти

Денна джоба (EOD ціни+курси) + кнопка ручного перерахунку. Сервіс — `arch §3.5`; таблиця — `arch §1.9`; ендпоінти — `arch §2 snapshots`.

### FR-39 — Денний автоснапшот
`dailySnapshot` — останній крок EOD-пайплайна (`arch §4.3`).

- [ ] Щодня (23:30 Kyiv, після цін/курсів/погашень) рахується снапшот на сьогодні: `computePortfolioState` + оцінка → upsert `net_worth_snapshots`.
- [ ] Снапшот зберігає `snapshotDate`, `baseCurrency`, `totalMinor`, `breakdown` (byAccount/byClass/byCurrency у базовій).
- [ ] Один рядок на (user, date); повторний запуск того ж дня — upsert (перезапис).
- [ ] Успіх → `app_settings['eod.lastSuccessDate'] = today`.
- [ ] Catch-up: якщо при старті бекенда `eod.lastSuccessDate` старіша за вчора — одноразовий запуск пайплайна через ~60с після boot (`arch §4`).

### FR-40 — Ручний перерахунок снапшота(ів)
`POST /api/snapshots/run {date?}` (один день); `POST /api/snapshots/rebuild {from, to}` (діапазон) (`arch §2`).

- [ ] Кнопка в UI запускає `run` на обрану/сьогоднішню дату → upsert снапшота.
- [ ] `rebuild {from,to}` послідовно перераховує діапазон зі збереженої історії цін/курсів; відповідь `{count}`.
- [ ] Ручний перерахунок використовується після редагування історичних транзакцій (бо автоперебудова минулого не відбувається, FR-22).
- [ ] `run` без `date` — сьогодні (Kyiv).

## Нефункціональні вимоги

### NFR-01 — Приватність: вичерпний список зовнішніх викликів
Дані лише на власній інфраструктурі; жодних аналітики/телеметрії/третіх сторін поза переліком.

| Напрям | Призначення | Контекст | Дозвіл |
|---|---|---|---|
| `query1/query2.finance.yahoo.com` | Котирування акцій/ETF/крипти | Рантайм (джоба `syncPrices`) | ✓ |
| `api.frankfurter.dev` / `api.frankfurter.app` | Крос-курси ECB | Рантайм (джоба `syncFxRates`) | ✓ |
| `bank.gov.ua` (НБУ) | Офіційний курс UAH | Рантайм (джоба `syncFxRates`) | ✓ |
| Telegram Bot API | Нотифікації деплою | CI/CD (не рантайм), мінімум даних | ✓ |

- [ ] Жодних інших вихідних мережевих викликів із рантайму бекенда (перевірка: код не містить fetch до інших хостів).
- [ ] Жодної фронтенд-аналітики/трекерів/CDN сторонніх (CSP `connect-src 'self' https://api.statok.simk.in.ua`, `arch §9`).
- [ ] Telegram-нотифікації несуть мінімум даних (статус/версія деплою), без вмісту портфеля.
- [ ] `docker.sock` у контейнери НЕ монтується (`arch §9`).

### NFR-02 — Безпека
Деталі — `arch §9`.

- [ ] Усі ендпоінти крім login/health — під JWT (CRR-1).
- [ ] Паролі — bcrypt (cost 10); у логи/відповіді не потрапляють.
- [ ] CORS — строгий allowlist із `CORS_ORIGINS` (прод — лише `https://statok.simk.in.ua`), `credentials: false`.
- [ ] Security headers на бекенді (`hono/secure-headers`) і nginx (CSP/HSTS/X-Frame-Options/nosniff, `arch §9`).
- [ ] Rate-limit логіну (FR-03); JWT-секрет ≥ 32 байти (валідація на старті).
- [ ] `pg_dump`-ендпоінт лише під auth; стрім без запису на диск.
- [ ] Секрети — лише на VPS (`/opt/statok/.env`), ніколи в репо.

### NFR-03 — Продуктивність
Один користувач, до ~20k транзакцій.

- [ ] Стандартні API-відповіді (positions, valuation, pnl, журнал, дашборди) < 500мс при ~20k транзакцій.
- [ ] Позиції/оцінка рахуються fold-ом на льоту (без кешу), індекси `arch §1.6` покривають вибірки.
- [ ] Журнал транзакцій пагінований (limit дефолт 50, макс 500); решта списків без пагінації прийнятна для single-user.
- [ ] Фронтенд-бандл компактний (uPlot ~12KB замість Chart.js, `arch §7.4`); первинне завантаження SPA швидке.

### NFR-04 — Надійність джоб (ретраї, ідемпотентність)
`arch §4`.

- [ ] Кожен HTTP-запит джоби має ретраї (`withRetry`: 3 спроби, 1s/4s/16s + джиттер).
- [ ] Фінальна невдача джоби логується в `app_settings['job.*'].lastError`, НЕ валить процес.
- [ ] EOD-пайплайн послідовний: помилка кроку логується, наступні кроки виконуються (снапшот — з наявних даних).

| # | Поза скоупом | Примітка |
|---|---|---|
| 1 | CSV-**імпорт** транзакцій | Лише експорт у v1; генерик CSV-маппер + пресети — після MVP (`requirements.md`). |

| 4 | Алокаційний дашборд / performance-дашборди | Лише net worth + cashflow; розрізи byClass/byAccount є в API, але окремого дашборда нема. |
| 5 | Цінові алерти / алерти дрейфу | Telegram — лише деплой-інфраструктура; продуктових алертів нема. |
| 6 | Податкові звіти | Лише realized P&L у базовій; денна історія НБУ зберігається для майбутнього річного звіту в UAH. |

| 8 | Multi-user / шаринг | Один сід-користувач; `user_id` присутній як заділ, але без UI/реєстрації. |
| 9 | In-kind перекази активів | Перекази — лише грошові ноги (FR-17). |
| 10 | Періодичні збори брокера | Комісія — лише поле `fee` на buy/sell; окремого типу «періодичний збір» нема. |
| 11 | CoinGecko / платні цінові API | Лише Yahoo + ручна ціна як фолбек. |
| 12 | Реінвест дивідендів (DRIP) автоматичний | Реінвест — вручну окремою buy-транзакцією (опційно, `requirements.md`). |

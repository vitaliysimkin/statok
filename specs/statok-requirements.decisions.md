# Decisions for Statok — вимоги та фічі

> Working file. Will be distilled into `requirements.md` and deleted.

**About:** Збір вимог до self-hosted застосунку особистих фінансів Statok. Фаза 1 — інвестиції/портфель, фаза 2 — витрати/аналітика. Файл фіксує продуктові вимоги і підтвердження стеку/деплою (патерн tardis). Набір переглянуто другою моделлю: додано комісії, корпоративні дії, перекази між рахунками, податковий контекст, бекапи, автентифікацію; дублікати злито. 30 питань.

## Design Decisions

### Q1: Скоуп MVP (фаза 1)?

**Options:**
- **A)** Тільки інвестиції/портфель: активи, транзакції, дохідність, net worth — ➕ швидкий вихід ➕ фокус ➖ без витрат
- **B)** Інвестиції + базовий ручний облік витрат одразу — ➕ повніше ➖ довше до першої користі
- **C)** Інвестиції + авто-синк API брокерів одразу — ➕ автоматизація ➖ складно, безпека ключів

**Recommendation:** A — спершу довести інвестиційне ядро; витрати — фаза 2 (Q17), API-синк — пізніше (Q6).

**Answer:** A
**Comment:** 
### Q2: [assumption] Single-user і приватність: один акаунт без шарингу, дані не покидають власну інфру.

**Options:**
- **A)** Підтверджую: єдиний користувач; зовнішні виклики лише до цінових/курсових API (Yahoo/CoinGecko/ECB/НБУ); Telegram-алерти надсилають мінімум (тікер/ціна/версія релізу)

**Recommendation:** A — multi-user/шаринг поза скоупом v1; якщо фінансові алерти через Telegram бентежать — зазнач у Comment.

**Answer:** A
**Comment:** 
### Q3: Платформа/доступ?

**Options:**
- **A)** Веб, десктоп-первинно, адаптив під мобільний браузер — ➕ просто ➖ не «апка»
- **B)** Веб + PWA (встановлюється на телефон, базовий офлайн-кеш) — ➕ телефонний досвід малою ціною ➖ трохи більше роботи
- **C)** Тільки десктоп-веб — ➕ мінімум ➖ незручно з телефону

**Recommendation:** B — PWA дешевий, а у фазі 2 стане основним способом вводу витрат з телефону.

**Answer:** B
**Comment:**

### Q5: Рахунки як перший клас?

Чи групуються позиції/транзакції по рахунках (брокер, біржа, банк, гаманець), чи портфель плаский.

**Options:**
- **A)** Рахунки як сутність: кожна транзакція й позиція належить рахунку — ➕ алокація по інституціях ➕ основа для переказів (Q9) ➖ +1 сутність в UI
- **B)** Плаский портфель без рахунків — ➕ простіше ➖ перекази й розбивка по брокерах неможливі, доробляти потім дорого
- **C)** Рахунок як необовʼязковий тег — ➕ компроміс ➖ напівзахід, цілісність не гарантована

**Recommendation:** A — дешево закласти на старті й дорого додавати потім; без цього Q9 не існує.

**Answer:** A
**Comment:** 
### Q6: Як вводяться позиції/транзакції?

**Options:** [combinable]
- **A)** Ручний ввід транзакцій (buy/sell/dividend) — ➕ контроль ➕ просто
- **B)** Імпорт CSV (брокер/біржа) — ➕ швидко завести історію
- **C)** API брокера/біржі (read-only ключі) — ➕ автосинк ➖ складно, безпека ключів
- **D)** Ручний «снапшот» балансу активу без історії угод — ➕ для неліквідних/депозитів

**Recommendation:** A,B,D — ручний + CSV + снапшоти на старт; API-синк (C) свідомо пізніше.

**Answer:** A
**Comment:** тільки для початку можливо додати якийсь "поточний стан"
### Q7: Комісії та збори?

**Options:** [combinable]
- **A)** Поле «комісія» на кожній угоді: включається в собівартість купівлі / зменшує виручку продажу — ➕ чесний P&L
- **B)** Окремий тип транзакції «збір» (депозитарні/щомісячні комісії, газ-фі поза угодами) — ➕ повна картина витрат на інвестування
- **C)** Без комісій у v1 — ➕ менше полів ➖ P&L систематично завищений

**Recommendation:** A,B — обидва типи реальні (разова комісія угоди + періодичні збори брокера).

**Answer:** A
**Comment:** 
### Q8: Корпоративні дії (спліти, зміна тікера)?

**Options:**
- **A)** Типи «спліт» (коефіцієнт перерахунку кількості) + «зміна тікера/перейменування» зі збереженням історії — ➕ покриває ~95% випадків
- **B)** Вручну: видалити й перезавести позицію — ➕ нуль коду ➖ ламає історію дохідності
- **C)** Повний модуль corp actions (мерджі, спін-офи, дивіденди акціями) — ➕ повнота ➖ великий скоуп

**Recommendation:** A — спліти трапляються регулярно (NVDA, TSLA…), без них історія цін/кількостей бреше.

**Answer:** A
**Comment:** не знаю що це але ок. В мене поки тільки ОВДП, без акцій
### Q9: Перекази між власними рахунками?

**Options:**
- **A)** Звʼязана пара transfer-out/transfer-in для грошей між рахунками — ➕ внески на брокера не виглядають як дохід/витрата, XIRR і кешфлоу чесні
- **B)** A + переказ активів in-kind (між брокерами/на холодний гаманець) зі збереженням собівартості — ➕ повнота ➖ рідкісний кейс
- **C)** Незвʼязані deposit/withdraw на кожному рахунку — ➕ просто ➖ дохідність і кешфлоу-звіти забруднені фейковими притоками

**Recommendation:** A — без звʼязаних переказів XIRR (Q14) бреше; B додати, коли реально переноситимеш активи.

**Answer:** A
**Comment:** 
### Q10: Базова валюта?

Архітектурно: позиції живуть у своїх валютах, net worth і звіти конвертуються в одну базову.

**Options:**
- **A)** UAH — ➕ рідна, збігається з податковою ➖ волатильна мірка для доларових активів
- **B)** USD — ➕ більшість портфеля доларова, стабільна мірка ➖ податки все одно в UAH
- **C)** EUR
- **D)** Базова + друга «звітна» валюта (головні цифри дублюються в обох) — ➕ найкраще з двох світів ➖ трохи більше UI

**Recommendation:** B — портфель здебільшого доларовий, а податковий звіт усе одно окремо в UAH за НБУ (Q20); хочеш бачити статок і в грн — обирай D.

**Answer:** B
**Comment:** частина рахунків в UAH, частина в USD, EUR
### Q11: Джерела ринкових цін (акції/ETF + крипта)?

**Options:** [combinable]
- **A)** Yahoo Finance (неофіційний API) — акції/ETF — ➕ безкоштовно, без ключа ➖ неофіційний, може ламатись
- **B)** Stooq — акції/ETF, EOD CSV — ➕ стабільний і простий ➖ покриття тикерів менше
- **C)** CoinGecko (free tier) — крипта — ➕ широке покриття ➖ rate limit
- **D)** Платний/ключовий API (Finnhub, Twelve Data, EODHD, Alpha Vantage) — ➕ стабільність ➖ ключ/$
- **E)** Ручне редагування ціни — універсальний фолбек — ➕ нуль залежностей

**Recommendation:** A,C,E — безкоштовного стека вистачить (B як запасний для акцій); D лише якщо впремось у ліміти.

**Answer:** A,E
**Comment:** 
### Q12: Деталізація облігацій?

**Options:** [combinable]
- **A)** Номінал, купон, частота виплат
- **B)** Дата погашення, автопогашення в кеш
- **C)** YTM / поточна дохідність (розрахунок)
- **D)** Достатньо як «актив із ціною», без облігаційної специфіки

**Recommendation:** A,B — для ОВДП/військових облігацій купон і погашення головне; C приємний бонус, не блокер.

**Answer:** A,B,C
**Comment:** 
### Q13: Дивіденди/купони?

**Options:**
- **A)** Виплати з полями брутто / утриманий податок / нетто + історія доходу + опційний реінвест (DRIP) — ➕ чиста дохідність із урахуванням withholding tax
- **B)** Тільки факт виплати (нетто-сума, дата) — ➕ простіше ➖ податок біля джерела губиться
- **C)** Поки не треба — ➕ менше скоупу

**Recommendation:** A — утримання 15–30% на дивіденди США суттєво міняє реальну дохідність, поля краще закласти одразу.

**Answer:** A
**Comment:** 
### Q14: Метрики дохідності?

**Options:** [combinable]
- **A)** Проста P&L (нереалізована/реалізована) + %
- **B)** Time-Weighted Return (TWR)
- **C)** Money-Weighted / XIRR
- **D)** Порівняння з бенчмарком (S&P 500 тощо)

**Recommendation:** A,C — P&L база, XIRR чесно враховує довнесення (паливо для нього — звʼязані перекази з Q9); TWR/бенчмарк пізніше.

**Answer:** A
**Comment:** ми ж пізніше зможемо додати, тому тут поки мінімум
### Q15: Метод обліку собівартості (realized gains)?

**Options:**
- **A)** Середньозважена (average cost) — ➕ просто і зрозуміло
- **B)** FIFO — ➕ ближче до податкових практик ➖ складніше (лоти)
- **C)** Обидва з перемикачем — ➕ гнучко ➖ подвійна робота

**Recommendation:** A для v1 — достатньо для особистого трекінгу; FIFO додамо, якщо податковий звіт (Q20) того вимагатиме.

**Answer:** A
**Comment:** 
### Q16: Історія net worth / вартості портфеля?

**Options:**
- **A)** Денні снапшоти фоновою джобою — ➕ гладкі графіки, швидкі звіти ➖ потрібен крон (патерн jobs/ у tardis вже є)
- **B)** Перерахунок на льоту з історії транзакцій + цін — ➕ без фону ➖ повільно, складно
- **C)** Снапшот при кожній зміні даних — ➕ просто ➖ нерівномірна крива

**Recommendation:** A — денна джоба тягне EOD-ціни/курси й пише снапшот; плюс кнопка ручного перерахунку.

**Answer:** A
**Comment:** поки не критично

### Q20: Податковий контекст (резидент України)?

**Options:**

- **C)** Повний податковий модуль (декларація, перенесення збитків) — ➖ великий скоуп, мінливе законодавство

**Recommendation:** A на старт, але схема одразу зберігає курс НБУ на дату угоди — тоді B зʼявиться без міграції даних.

### Q23: Бекапи та відновлення?

### Q24: Автентифікація (single-user)?
